PHASE 2.7 PROMPT 3 — Build safe scheduled data pipeline framework for Kenyan investment sources

Now build the first version of scheduled data pipelines for PesaRoute.

Goal:
Create a safe, auditable, slow-growing data pipeline framework that can regularly update the Kenya Investment Knowledge Base from public/official sources.

Important:
Do not build fragile scraping everywhere.
Do not bypass robots.txt or terms.
Do not redistribute restricted NSE market prices.
Do not add AI.
Do not add investment execution.
Do not add account linking.
Do not create data that looks verified if it has not passed validation/review.

Pipeline philosophy:
Extract safely → store raw snapshot → parse → normalize → validate → stage → compare changes → admin review if needed → publish canonical data.

Pipeline stages:

1. source discovery/check
2. fetch raw content
3. store raw snapshot
4. parse into SourceRecord
5. normalize
6. validate
7. create DataQualityIssue if needed
8. upsert staging data
9. compare with canonical data
10. auto-approve low-risk updates
11. require admin review for sensitive changes
12. publish
13. update freshness status
14. notify admin of important changes

7. Add freshness logic:
   For each ProductPassport, Provider, RegulatedEntityStatus, ListedCompany, SaccoEntity, GovernmentSecurityReference:

* fresh if last_verified_at within expected update frequency
* acceptable if slightly older
* stale if older than threshold
* unknown if never verified

Suggested thresholds:

* CBK auction data: stale after 14 days
* NSE listed company list: stale after 30 days
* CMA licensees: stale after 30–60 days
* SASRA SACCO lists: stale after 90 days or annually depending source
* RBA/IRA: stale after 60–90 days
* generic educational content: review every 180 days
